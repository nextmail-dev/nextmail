# 第十三阶段：签名重构、账户管理与通知

## 状态

实施中。第一批“默认签名与自动插入”和第二批“账户下拉菜单管理入口”已于 2026-07-22 通过 Windows 手动验收；第三批“分层通知偏好与新邮件判定”正在实施。

本阶段承接第十二阶段验收后的产品调整。优先完成明确功能，不夹带会话、托盘、OAuth、POP3、发布硬化或全局优化。第十一阶段已经验收的本地全文搜索保持不变，原定于第十一阶段的通知范围改由本阶段实施。

## 已确认产品行为

### 签名

- 模板继续保留新建、回复、回复全部和转发四场景规则；签名不再按四个场景分别配置，而是每个定义范围只有一个默认签名。
- 全局范围和账户范围继续沿用第九阶段的显式作用域。账户没有自己的签名偏好时继承全局默认；账户首次创建自己的签名时，该签名自动成为该账户默认。
- 每个签名在列表中提供“设为默认”，当前默认项有明确标记。创建某范围的第一个签名后自动设为默认。
- “自动为邮件选择默认签名”决定首次创建新建、回复、回复全部或转发草稿时是否插入已解析的默认签名。用户在 Composer 中主动切换或删除签名的既有语义不变，保存或重开不会重新插入。
- 迁移会从既有场景签名规则中按新建、回复、回复全部、转发的优先顺序保留一个默认值，随后由新的签名偏好成为唯一自动选择来源；模板规则不受影响。

### 账户管理

- 左上角账户身份始终是可打开的下拉菜单，即使当前只有一个账户。
- 菜单上部保留账户切换，底部通过分割线提供“账户管理”。账户管理承载添加、连接编辑、重新认证、同步策略、文件夹映射和安全移除；设置侧栏不再保留重复入口。
- 账户管理复用现有窄 API、TanStack Query 和表单组件，不增加 React 对凭据库、SQLite 或邮件服务器的直接访问。

### 新邮件通知

- 设置提供全局通知开关、逐账户开关和逐文件夹开关；文件夹覆盖项只保存公开账户/文件夹 ID，不保存邮件正文、凭据、服务器错误或内部路径。
- 通知内容仅展示账户身份、发件人和主题，不展示正文预览。首次全量同步和静默缓存重建不得把历史邮件误报为新邮件；同一账户/文件夹/消息必须去重。
- 展示模式为“层叠”或“覆盖”。层叠模式可配置最大同时显示数量；覆盖模式只保留一个通知窗口并以新邮件替换旧内容。展示时间默认 5 秒。
- 通知使用 NextMail 自有的无装饰桌面窗口放在当前工作区右下角，以满足层叠数量、覆盖和展示时间的确定性要求；不把这些行为伪装成操作系统通知中心能力。窗口不获得数据库、文件系统或通用系统 API 权限。
- 点击通知聚焦主窗口并定位到账户、文件夹和消息。目标已移动或删除时，降级为打开对应账户/文件夹，不报错也不跨账户猜测。
- Windows 10 22H2+ 是本阶段当前实机门禁；macOS 12+ 的工作区定位、Spaces/多显示器和通知窗口层叠必须实际执行后才能声明通过。

## 实施批次

### 第一批：默认签名与自动插入

状态：已于 2026-07-22 通过 Windows 手动验收。

- 新增数据格式版本 17 的签名偏好迁移、稳定 DTO/Command/API 与账户槽隔离的 Repository 方法。
- 重构写信设置中的签名列表，增加默认标记、“设为默认”和自动选择复选框；四场景卡片只管理模板。
- Composer 首次组装从签名偏好解析默认签名，继续由 Rust 渲染变量并生成稳定 `nextmailSignature` 节点。
- 覆盖首次签名自动默认、全局继承/账户覆盖、关闭自动插入、陈旧 revision 和前端交互回归。

### 第二批：账户下拉菜单管理入口

状态：已于 2026-07-22 通过 Windows 手动验收。

- 提取现有账户列表和管理面板供主工作区复用；账户下拉菜单始终可用并在底部加入账户管理入口。
- 移除设置侧栏的重复账户类别，但保留无账户时的正式添加入口和首次启动账户流程。
- 验证多账户切换、添加、删除当前/最后账户、窗口焦点、键盘导航和 Query 失效。

### 第三批：分层通知偏好与新邮件判定

状态：实施中。

- 建立注入式通知偏好存储和稳定 API，完成全局、账户、文件夹三级开关及模式、数量、展示时间设置 UI。
- 在同步提交结果中形成只含公开账户/文件夹/消息 ID、发件人和主题的新邮件候选；建立基线、去重与首次同步抑制，不持有 SQLite 写锁等待窗口操作。

### 第四批：通知窗口、层叠/覆盖与点击定位

状态：尚未实施，第三批验收后开始。

- 实现受限通知 WebView、右下角布局、层叠上限、覆盖更新、超时关闭和主窗口定位。
- 补 Windows 多显示器、窗口隐藏/最小化、连续到信、邮件已移动/删除和应用退出清理回归；为 macOS 保留明确实机门禁。

## 安全与架构边界

- 签名富文本继续通过现有 Rust 清洗、变量白名单和稳定节点进入草稿；迁移不把定义内容复制到新表。
- 所有账户级签名偏好和通知判定都按 `account_slot_id` 或经验证的公开账户 ID 隔离；React 不接触匿名槽。
- 通知窗口只接收最小展示 DTO，不接收原始 EML、正文 HTML、附件路径、密码、Token 或任意 URL。
- 不新增忽略 TLS、远程图片放行、HTML sandbox 权限或通用文件/网络 Capability。
- 不运行 Tauri bundle，不提交未经用户实机确认的批次。
