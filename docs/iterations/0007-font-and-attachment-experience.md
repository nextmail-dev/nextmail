# 第七阶段：系统字体与附件体验优化

状态：已验收。

日期：2026-07-18

## 一、阶段目标

在开始 POP3 之前进行一轮范围受控的桌面体验优化：移除当前随包的 Roboto 与 Droid Sans Fallback，改用 Windows/macOS 原生字体栈验证不同字号下的清晰度；压缩邮件正文附件区域的视觉占用，并建立“按需下载后使用系统默认程序打开”的完整附件交互。

本阶段只调整应用界面字体和已收邮件附件，不改变邮件正文原始排版、写信附件、IMAP 同步语义、账户模型或协议能力。

## 二、系统字体试验

### 字体策略

- 应用 UI 改用平台原生字体栈，不再要求字体随安装包分发。
- Windows 优先使用系统 UI 与雅黑字形：`system-ui`、`Segoe UI`、`Microsoft YaHei UI`、`Microsoft YaHei`、`sans-serif`。
- macOS 优先使用系统 UI 与苹方字形：`-apple-system`、`BlinkMacSystemFont`、`SF Pro Text`、`PingFang SC`、`sans-serif`。
- 其他平台仅使用 `system-ui` 与 `sans-serif` 作为非验收回退，不为 Linux 单独适配。
- 本轮保留现有 Windows 12/14px、macOS 11/13px 的字号令牌、主题对比度和平台平滑设置，只替换字体来源，以便单独判断字体变化的实际效果。
- 应用 UI、标题栏、设置和写信窗口共用同一平台字体栈；安全 HTML 邮件继续保留自身经过清洗的字体声明，不强制覆盖为应用字体。

### 依赖与文档

- 移除 `src/main.tsx` 中的 Roboto CSS 导入和 `base.css` 中的 `NextMail CJK` 字体声明。
- 使用 pnpm 移除 `@fontsource/roboto` 与 `@fontpkg/droid-sans-fallback`，同步更新 `pnpm-lock.yaml`。
- 确认发布包不再包含字体文件后，删除仅为这两套字体保留的许可证资源，并更新 `third-party-notices.md`、架构基线和变更记录。
- 不在本阶段引入另一套第三方字体；若系统字体仍不满足 Windows 实机效果，再另行评估字体方案。

## 三、附件区域视觉调整

- 邮件正文与附件区域之间增加明确但紧凑的顶部间距，避免当前顶部贴合。
- 减少附件区域左右、底部外边距和内部 padding，降低只有一两个附件时的整体高度。
- 标题、附件名、大小和状态保持清晰层级；长文件名截断但通过提示保留完整名称。
- 附件列表继续自动换行，按钮维持足够的键盘焦点和点击范围，不使用原生链接或原生按钮样式。
- 调整只作用于阅读器附件区域，不改变写信窗口已选附件的布局。

建议初始视觉值为：顶部间距 8px、左右外边距 20px、底部外边距 16px、容器内边距 12px；实机验收时再根据整体密度微调。

## 四、附件下载与打开闭环

### 产品行为

- “下载后自动打开”作为设备级阅读偏好提供，默认开启并允许用户关闭。未下载附件点击后显示加载状态；开启时由 Rust 按需取得内容并在同一次操作中使用系统默认程序打开，关闭时首次点击只下载，后续点击仍可手动打开。
- 已下载附件不再禁用，后续点击直接从本地内容存储生成或复用可打开缓存文件并打开。
- 下载或打开失败时显示稳定、本地化的错误信息；前端不接收内部路径、内容哈希、服务器响应或系统命令文本。
- 同一附件正在下载或打开时禁止重复触发；不同附件不共享错误和加载状态。
- 每个附件提供独立“另存为”入口，由系统保存对话框选择目标位置；本阶段不增加通用下载管理器、附件预览器、“全部下载”或写信附件打开功能。

### Rust 边界

- 保留现有 `request_attachment(account_id, attachment_id)` 作为纯下载能力，并增加面向产品意图的窄命令，例如 `open_message_attachment(account_id, attachment_id)`。
- 命令必须先验证 `account_id -> data_slot_id -> message -> attachment_id` 所属关系；不能通过其他账户的附件 ID 读取或打开内容。
- 内容不可用时复用现有 IMAP 按需下载流程；内容可用时直接读取内容寻址存储，不重复访问服务器。
- 在可重建的 `cache/` 下按不透明附件 ID 建立打开副本，文件名经过跨平台清理并保留安全扩展名，不能出现路径穿越、绝对路径、Windows 保留名或控制字符。
- 系统打开能力封装在 Rust Adapter 中，只允许打开本命令刚刚创建并验证的缓存文件；React 和 Tauri Capability 不获得任意文件、Shell 或任意路径打开权限。
- 自动测试使用假的系统打开 Adapter，不实际启动外部程序；临时缓存目录在测试结束后删除，正式应用缓存继续按既有策略保留为可重建数据。

### 高风险附件行为

常规附件交给系统默认程序打开。可执行文件、脚本、快捷方式和其他高风险扩展名不直接执行，也不弹出执行确认；用户触发打开时只在系统文件管理器中显示缓存文件。用户仍可通过明确的“另存为”把文件复制到自己选择的位置。

## 五、公共命令与错误

预计新增或调整：

- `open_message_attachment(account_id, attachment_id)`
- `AttachmentOpenStatus`：仅在确有必要时返回 `downloading | opening | opened` 等稳定状态；常规进度优先由前端 mutation 表达。
- 稳定错误码：附件不存在、内容下载失败、缓存文件创建失败、文件名不可用、系统没有关联程序、系统打开失败和高风险附件未确认。

不增加接收任意文件路径的公共命令，也不把 `attachments/` 或 `cache/` 的真实位置返回前端。

## 六、测试与验收

### 自动验证

- 前端组件测试覆盖未下载附件、已下载附件、加载防重、错误提示和键盘触发。
- Rust 单元测试覆盖账户隔离、按需下载后打开、已缓存直接打开、文件名清理、缓存复用和系统打开失败。
- 使用临时数据目录和假的系统打开 Adapter，不在测试期间启动外部应用或写入真实用户下载目录。
- 执行 TypeScript、前端测试、Rust 测试、Clippy、格式检查和 Vite 构建；按约定不执行完整 Tauri bundle。

### 手动验收

- Windows 10/11 在 100%、125%、150% 缩放下检查中英文标题、菜单、邮件列表、设置和写信窗口，各字号应清晰且不出现缺字或明显布局跳动。
- macOS 检查英文与中文是否正确使用系统 UI/苹方回退，视觉密度不应明显劣于当前版本。
- 浅色和深色模式均检查附件区域：顶部不贴合、四周留白紧凑、长文件名和多个附件不破坏布局。
- 未下载附件点击后可以完成下载并自动打开；已下载附件再次点击无需联网即可打开。
- 文本、图片、PDF、Office 文档及没有系统关联程序的文件均有可理解结果；文件名包含中文、空格和特殊字符时仍能安全工作。
- 附件操作不暴露内部路径，不允许跨账户附件 ID，也不能打开用户未选择的任意本机文件。

## 七、实施顺序

1. 增加系统打开端口、文件名清理和缓存副本测试，确定平台实现与高风险附件规则。
2. 实现账户隔离的附件打开用例和窄 Tauri Command，复用现有按需下载与内容存储。
3. 调整阅读器附件 mutation、可访问状态和紧凑布局，补齐中英文文案与组件测试。
4. 切换到平台系统字体栈，移除字体依赖与对应发布资源；不同时改变字号令牌。
5. 更新架构、第三方资源说明和功能变更记录，完成自动验证后交由用户手动验收。

## 八、明确非目标

- POP3、OAuth、模板与签名、全文搜索、托盘和通知。
- 内嵌 `cid:` 图片、远程附件代理、附件缩略图、应用内 PDF/Office 预览、杀毒扫描和云附件。
- 通用下载中心、“全部下载”、自动打开所有附件或把附件路径暴露给前端。
- 重新设计写信编辑器、写信附件或邮件正文 HTML 清洗策略。

## 九、实施结果

- 应用 UI 已切换到 Windows Segoe UI/微软雅黑与 macOS 系统 UI/苹方字体栈，随包字体依赖、CSS 导入和专用许可证资源已移除；既有平台字号令牌保持不变。
- 阅读偏好已增加默认开启的“下载附件后自动打开”，旧偏好文件缺失该字段时安全迁移为开启。
- 阅读器附件区域已改为紧凑布局；附件主操作支持按需下载和再次打开，每个附件都有独立“另存为”入口。
- Rust 已增加账户隔离的附件准备、系统打开和另存为用例；缓存副本使用安全文件名与不透明目录，高风险扩展名只在系统文件管理器中显示。
- 前端未获得 opener、Shell、任意文件或任意路径权限；Tauri Command 仍只接收账户 ID 与附件 ID。
- 自动检查已执行，结果记录于 `changes/0026-system-font-and-attachment-experience.md`；完整 Tauri 构建按约定未执行。
- 2026-07-18：系统字体、附件紧凑布局、自动打开偏好、另存为和高风险文件夹定位通过用户实机验收。

## 十、已经确认的产品决策

以下决策已由用户确认：

1. 普通附件支持“下载后自动打开”，并在阅读设置中允许关闭。
2. 每个附件提供独立“另存为”入口。
3. 高风险附件的打开动作改为在系统文件管理器中显示文件，不直接执行。
4. 使用“系统 UI 拉丁字体 + 微软雅黑/苹方中文回退”的平台字体栈。
