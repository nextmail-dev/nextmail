# 第十一阶段：搜索、会话与桌面集成

状态：未排期。等待第九、十阶段完成后再细化，尚未实施。

## 阶段定位

在模板/签名和邮件阅读/回复体验稳定后，为本地邮件数据补齐全文检索、会话聚合和桌面常驻能力。当前只保留高层范围，不提前决定 schema、系统权限或通知产品行为。

## 目标范围

- 使用 SQLite FTS5 搜索已同步到本机且允许索引的主题、地址、正文和必要元数据。
- 基于标准线程头和可解释的降级规则聚合会话，并保持账户、文件夹和规范邮件位置语义。
- 增加托盘、未读数、系统新邮件通知和经过确认的窗口行为。
- 搜索、会话和通知事件继续只传 ID/修订号，React 通过稳定 Query key 重读本地视图。

## 实施前必须确认

- 单账户搜索还是跨账户搜索、统一收件箱是否进入范围。
- FTS 索引哪些正文表示、如何处理 HTML、附件名、语言分词、重建和数据目录迁移。
- 会话主键、跨文件夹位置、Sent/Drafts 和缺失/畸形线程头的降级规则。
- Windows/macOS 托盘、关闭、后台运行、通知隐私和点击跳转行为。
- 通知预览是否显示发件人/主题，以及锁屏和系统权限策略。

## 安全与数据边界

- FTS 数据继续位于可迁移 SQLite，所有查询按 `account_slot_id` 隔离。
- 系统通知不得包含密码、Token、内部路径、服务器原始错误或未确认展示的敏感正文。
- 托盘和通知只通过窄 Tauri Command/Event 与平台 Capability 接入，不向 React 开放通用系统 API。

## 非目标

- 不实现联系人、规则、日历、PGP/S-MIME、企业策略或遥测。
- POP3、OAuth、发布硬化和全局优化仍为未排期设想。
- 不在第九、十阶段完成前提前创建迁移、Capability、后台常驻任务或 UI 占位。
