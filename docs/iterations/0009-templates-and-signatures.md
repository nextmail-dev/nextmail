# 第九阶段：模板与签名

状态：规划中。下一功能阶段，尚未实施。

## 阶段定位

在现有多账户、富文本草稿、回复/回复全部/转发和持久化发件基础上，补齐可管理、可复用的邮件模板与签名系统。该阶段优先完成功能，不承接旧重构计划或全局性能清理。

## 当前事实

- 写信窗口已经保存 Tiptap JSON、HTML 和纯文本三种表示。
- 当前工具栏只能临时插入由账户显示名和邮箱组成的基础签名，没有签名库、默认规则或可替换节点。
- 新建、回复、回复全部和转发已经可以创建草稿，但没有模板选择、变量替换或场景级默认签名。
- 草稿、邮件和附件均按 `account_slot_id` 隔离；密码与 Token 仍只允许进入系统凭据库。

## 目标

- 管理邮件模板和签名，并支持明确的全局/账户作用域。
- 为新建、回复、回复全部和转发分别选择默认模板与签名规则。
- 使用受控变量表生成主题和正文，变量缺失时有稳定、可本地化的结果。
- 在 Tiptap 文档中使用可识别的签名节点，切换身份、模板或签名时只替换对应节点，不破坏用户正文。
- 保持 editor JSON、HTML、纯文本和最终 MIME 的一致性。

## 计划范围

### 数据与命令

- 在正式实施前确认模板/签名的迁移语义、作用域优先级、排序和删除行为，再新增只位于 `src-tauri` 的 SQLite 迁移与窄 Repository。
- React 只通过 `src/app/api.ts` 和稳定 DTO 管理模板、签名及场景规则，不直接访问 SQLite。
- 所有账户级记录通过匿名账户槽隔离；模板正文和签名内容不得包含账户密码、Token 或内部路径。

### 变量与渲染

- 建立显式变量白名单、类型和转义规则，不执行脚本、表达式、任意模板代码或网络请求。
- HTML、纯文本和主题分别按上下文转义；未知变量不得静默生成危险 HTML。
- 变量替换在 Rust application 用例完成，Repository 只保存定义和结果。

### 写信体验

- 设置窗口提供模板、签名和默认场景规则管理；新增中英文生产文案和键盘可访问交互。
- 写信窗口可以选择模板/签名，并明确区分用户正文、模板正文和签名节点。
- 自动保存、草稿 revision、远端 Drafts、发件队列、回复线程头和转发附件语义保持不变。

## 实施前需要确认的产品决策

- 模板和签名是否允许全局共享，以及账户级规则覆盖全局规则的准确优先级。
- 默认签名放置位置、回复/转发是否自动插入，以及用户手动删除后是否再次自动恢复。
- 第一批变量清单及日期、称呼、账户身份等变量的格式化规则。
- 删除正在被默认规则引用的模板/签名时采用阻止、解除引用还是替换策略。

上述决策未确认前，本阶段保持“规划中”，不开始 schema 或 UI 实施。

## 非目标

- 不实现第十阶段的邮件 `<style>` 保真、外链确认、深色正文和完整 HTML 引用。
- 不实现全文搜索、会话、托盘、系统通知、POP3 或 OAuth。
- 不引入商业 Tiptap 扩展、云模板服务或远程模板市场；非宽松许可证依赖必须先确认。
- 不进行全局性能、lint/format、CI 或主包拆分清理。

## 验收门禁

- 自动测试覆盖作用域优先级、变量转义、签名节点可靠替换、四种写信场景和三格式 round-trip。
- Windows 实机验证多账户模板/签名隔离、草稿重开、远端 Drafts、发送与重启持久化。
- 完成自动验证后状态改为“等待手动验收”；用户确认前不提交或进入第十阶段。
