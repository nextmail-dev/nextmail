# 第九阶段：模板与签名

状态：实施中。第一批“模板与签名库”已验收（2026-07-20）；第二批“场景规则与 Composer 集成”待实施。

## 阶段定位

在现有多账户、富文本草稿、回复/回复全部/转发和持久化发件基础上，补齐可管理、可复用的邮件模板与签名系统。该阶段优先完成功能，不承接旧重构计划或全局性能清理。

## 当前事实

- 写信窗口已经保存 Tiptap JSON、HTML 和纯文本三种表示。
- 当前工具栏只能临时插入由账户显示名和邮箱组成的基础签名，没有签名库、默认规则或可替换节点。
- 新建、回复、回复全部和转发已经可以创建草稿，但没有模板选择、变量替换或场景级默认签名。
- 草稿、邮件和附件均按 `account_slot_id` 隔离；密码与 Token 仍只允许进入系统凭据库。

## 目标

- 管理邮件模板和签名，并支持明确的全局/账户作用域。
- 为新建、回复、回复全部和转发分别选择默认模板与签名规则。
- 使用受控变量表生成主题和正文，变量缺失时有稳定、可本地化的结果。
- 在 Tiptap 文档中使用可识别的签名节点，切换身份、模板或签名时只替换对应节点，不破坏用户正文。
- 保持 editor JSON、HTML、纯文本和最终 MIME 的一致性。

## 计划范围

### 数据与命令

- 在正式实施前确认模板/签名的迁移语义、作用域优先级、排序和删除行为，再新增只位于 `src-tauri` 的 SQLite 迁移与窄 Repository。
- React 只通过 `src/app/api.ts` 和稳定 DTO 管理模板、签名及场景规则，不直接访问 SQLite。
- 所有账户级记录通过匿名账户槽隔离；模板正文和签名内容不得包含账户密码、Token 或内部路径。

### 变量与渲染

- 建立显式变量白名单、类型和转义规则，不执行脚本、表达式、任意模板代码或网络请求。
- HTML、纯文本和主题分别按上下文转义；未知变量不得静默生成危险 HTML。
- 变量替换在 Rust application 用例完成，Repository 只保存定义和结果。

### 写信体验

- 设置窗口提供模板、签名和默认场景规则管理；新增中英文生产文案和键盘可访问交互。
- 写信窗口可以选择模板/签名，并明确区分用户正文、模板正文和签名节点。
- 自动保存、草稿 revision、远端 Drafts、发件队列、回复线程头和转发附件语义保持不变。

## 已确认的产品语义

- 模板和签名允许全局共享，也允许归属于单个账户。账户场景规则显式引用定义，不依赖名称覆盖；解析默认规则时账户规则优先于全局规则。
- 新建、回复、回复全部和转发可以分别配置模板与签名。签名只在草稿首次创建或用户主动切换时插入；用户手动删除后不会被自动保存或重开流程恢复。
- 第一批变量白名单为 `sender_name`、`sender_email`、`recipient_name`、`recipient_email` 和 `date`。未知变量在保存模板时拒绝；上下文缺失时插入操作返回稳定错误，不静默删除变量。`date` 使用本机日期并按当前界面语言格式化。
- 被全局或账户默认规则引用的模板/签名禁止删除；用户需要先解除引用。这样不会在删除时隐式改变其他账户或写信场景。
- 定义列表按名称进行稳定排序，不在本阶段增加手动拖拽排序。

## 第一批实施结果

- 新增数据格式版本 8：`mail_templates` 与 `mail_signatures` 以空数据槽表示全局作用域，以匿名 `account_slot_id` 隔离账户作用域，并保存 editor JSON、HTML、纯文本、revision 与更新时间。
- 新增 `CompositionDefinitionRepository`；所有 SQLx 类型留在 storage，Composer runtime 只用公开账户 ID 解析数据槽并调用窄方法。
- 新增模板/签名各四个稳定 Command 与前端 API：按作用域列出、新建、更新和带 revision 删除。application 统一规范名称和模板主题，并验证三格式大小与 editor JSON。
- 设置窗口“写信”分类提供全局/账户范围选择、模板和签名列表、复用 Tiptap 的富文本新建/编辑、二次确认删除及中英文生产文案。
- 当前批次没有把定义插入新建/回复/回复全部/转发草稿，现有临时基础签名按钮、草稿、远端 Drafts、MIME 和发件行为保持不变。
- Rust 回归覆盖全局/账户隔离、跨账户不可见、持久化重开、revision 冲突和输入校验；前端回归覆盖全局创建、账户范围切换和二次删除确认。
