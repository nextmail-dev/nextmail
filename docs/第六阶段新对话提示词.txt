二、产品与平台基线

- 产品名为 NextMail，是 Tauri 2 + React 19/TypeScript + Rust 的跨平台桌面多协议邮件客户端。
- 首发验收平台：Windows 10 22H2+ x64、macOS 12+ Intel/Apple Silicon。
- Linux 不做深度适配，也不是当前验收平台。
- 当前已完成并验收：首次启动、单账户 IMAP 本地阅读、SMTP 写信/草稿、完整 IMAP 同步语义、回复/回复全部/转发、跨平台窗口壳、现代 SaaS UI、独立设置窗口、远程图片偏好、文件夹树和启动响应优化。
- 当前阶段只实现多账户 IMAP + SMTP 密码账户。
- 本阶段不要实现 POP3、OAuth、统一收件箱、跨账户搜索、账户拖动排序、身份/别名、联系人、模板/签名扩展、托盘、通知聚合或后续阶段内容。

三、不可破坏的总体架构边界

- React 不得直接连接 SQLite、邮件服务器、任意文件系统或系统凭据库。
- 所有网络、数据库、凭据、文件存储、安全渲染和后台 Worker 均位于 Rust。
- React 只读取 Rust 返回的本地数据视图；所有写操作通过稳定业务命令完成。
- 同步必须先落 SQLite，再发送只含账户、文件夹、消息 ID 和修订号的窄事件；前端收到事件后失效对应 TanStack Query，不通过事件推送完整邮件正文或账户配置。
- 第三方协议类型不得泄漏到领域层、公共 DTO 或前端。禁止暴露 SQLx、async-imap、lettre、Tauri 内部类型。
- 命令统一使用稳定错误 code、本地化参数和可选重试语义。不得向 UI、日志或错误详情暴露密码、Token、服务器原始响应、系统凭据引用或内部数据路径。
- 功能依赖优先使用 MIT、Apache、BSD、ISC 等宽松许可证。新增其他许可证前必须向我确认；不得引入商业组件、Pro、Cloud 或付费扩展。

Rust 当前只有 `src-tauri` 一个 Cargo package：

- `src-tauri/src/core`：领域 DTO、稳定错误、ports，不依赖 Tauri、SQLx 或具体协议库。
- `src-tauri/src/storage`：SQLx Repository 和内容寻址文件存储；迁移位于 `src-tauri/migrations`。
- `src-tauri/src/protocols`：IMAP/SMTP/MIME/HTML 清洗及后续协议 Adapter。
- `src-tauri/src/application`、`adapters`、`commands` 与运行时模块：Tauri Command/Event、Keyring、应用服务、窗口和后台运行时装配。

唯一的 `Cargo.toml`、`Cargo.lock` 和 `target` 均位于 `src-tauri`。不要在仓库根目录重新创建 Cargo Workspace、Cargo manifest、lockfile、`target` 或 `crates/`。继续按照依赖方向拆分，不要把所有多账户逻辑堆进单个文件，也不要为了形式把每个小功能拆成新的 crate。

四、数据、凭据和安全约定

用户选择的可迁移数据目录包含：

- content.sqlite
- raw/
- attachments/
- cache/
- .nextmail-data.json

数据目录外的系统应用配置区保存：

- 当前数据目录路径
- 账户邮箱地址、显示名称、服务器、端口、TLS、认证方式
- account_id 到匿名 data_slot_id 的映射
- 主题、语言、窗口、本机阅读偏好、最近选择账户等本机设置

密码只进入 Windows Credential Manager/macOS Keychain。不得降级为明文文件、SQLite、localStorage、sessionStorage 或前端 Zustand 持久化。

必须继续满足：

- 数据目录中不能新增账户邮箱、服务器、端口、密码、Token 或本机账户映射。
- 邮件正文和邮件头天然可能包含邮箱地址，这不等同于账户配置泄漏。
- 所有 Repository 访问必须用 account_id 解析到 data_slot_id，并验证消息、文件夹、附件、草稿、发件任务和待办操作属于该账户。
- 跨账户传入其他账户的 ID 必须返回 not_found，不能读取或修改数据。
- 不得清空、迁移、删除或重建我的真实数据目录，除非我明确要求。
- 不要读取、打印、固化或提交我的真实密码、邮箱凭据、服务器响应和系统凭据内容。

账户连接安全：

- IMAP/SMTP 支持 TLS、STARTTLS 和明确确认后的非加密连接。
- TLS 严格校验系统信任链，不提供常规“忽略证书错误”。
- IMAP 测试只登录并读取 Capability，不修改邮箱。
- SMTP 测试只连接和认证，不发送测试邮件。
- IMAP 和 SMTP 都通过后才能保存账户或配置更新。
- 任何保存步骤失败都必须进行补偿，不留下可见半成品账户或明文秘密。

邮件安全：

- MIME 和 RFC 2047 继续使用启用 full_encoding 的 mail-parser，不另写脆弱的字符集解析器。
- HTML 邮件继续在 Rust 白名单清洗并在 sandbox iframe 中显示；不得为了多账户功能放宽 scripts、forms、same-origin、top-navigation 或危险 URL 限制。
- 远程图片默认阻止，是否自动加载继续使用现有本机阅读偏好。

六、启动与后台同步约定（必须保留）

当前已经实现并验收“首帧优先”：

- Tauri setup 只装配 AppState，不抢跑邮件同步和发件 Worker。
- 主工作区完成至少一个绘制周期后，通过 start_background_services 启动后台服务。
- 启动先显示本地 SQLite 数据，不等待网络。
- 启动同步显示进度；自动 IDLE/轮询和后台同步保持无感，不展示进度条。
- 文件夹元数据落库后立即通知前端；其后每批最多 100 封摘要落库后刷新列表，实现渐进显示。
- MIME 解码、预览、附件分析和 HTML 清洗在顺序提交的 Tokio blocking worker 中执行，避免阻塞异步调度线程。

第六阶段把单一 MailRuntime 改为 Supervisor Registry 时，必须保留以上行为：

- UI 首帧前不得启动所有账户同步。
- 首帧后恢复所有账户的后台 Supervisor。
- 当前账户的启动/手动同步可以展示进度；非当前账户的后台同步不得打断阅读或弹出进度。
- 切换账户只读取目标账户本地 SQLite 视图，不等待网络连接。
- 同步继续先落库、后发窄事件，并逐批呈现。
- React Strict Mode 或重复启动命令不得创建重复 Supervisor/SendWorker。

七、第六阶段已经确认的产品决策

必须完整遵循 docs/iterations/0006-multi-account.md。下面是不能自行改变的摘要：

账户身份与重复：

- account_id 是本机配置稳定 ID；data_slot_id 是可迁移数据目录中的匿名隔离 ID，二者不能混用。
- 同一邮箱可用于不同服务器或用户名。
- 完全相同的邮箱地址、IMAP 主机、端口和登录用户名组合不得重复添加。
- 账户按创建时间保持稳定，本阶段不做拖动排序。
- 最近选择账户保存在系统应用配置区；丢失时回退到第一个账户。

账户移除：

- 移除账户会停止该账户后台任务、关闭对应写信窗口、删除本机账户配置并清理系统凭据。
- 匿名数据槽、邮件、附件和普通草稿默认保留，不做破坏性数据删除。
- 有正在发送的任务或仍需写回服务器的 pending operation 时阻止移除，并给出清晰原因。
- 仅有普通本地草稿不阻止移除。
- 凭据清理失败不恢复已经移除的账户；保存不含邮箱和秘密的清理引用并在后续重试。

编辑和重新认证：

- 编辑连接可以预填邮箱、名称和服务器配置，永远不回填密码。
- 配置或密码变更必须重新通过 IMAP 和 SMTP 双测试后才原子生效。
- 凭据缺失或服务端明确认证失败才进入 reauth_required。
- 网络、TLS 和临时服务错误不能误判为 reauth_required。
- reauth_required 停止无意义的自动认证重试，但本地邮件保持可读；重新认证成功后无需重启即可恢复。

Supervisor Registry：

- SupervisorRegistry 维护 account_id -> AccountSupervisor。
- 每个账户独立拥有唤醒、退避、启动同步状态、待办重放和 IDLE 生命周期。
- 共享 MailRepository/SQLite 连接池，依靠 account_slot_id 隔离，不为每个账户重复打开数据库连接池。
- 全局最多两个主动 IMAP 同步/写操作；同一账户保持串行。
- IDLE 等待不占用主动同步名额。
- 手动收取当前账户有较高优先级，但必须公平轮转，不能让后台账户长期饥饿。
- 新增、更新、重新认证和移除账户都在运行期协调，不要求重启应用。
- 移除时取消该账户 IDLE/退避，协议步骤最多等待 10 秒有界收尾；迟到事件通过账户代次丢弃。

发件：

- 账户内 FIFO，账户间公平调度。
- 全局最多同时发送两封；同一账户一次一封。
- 一个账户的超时、断网或认证失败不能阻塞其他账户。
- 每个 send_job 继续绑定创建时的 account_slot_id 和不可变 MIME。

配置事务：

- accounts.json 升级为并发安全的多账户集合，增加单调修订号和待清理凭据引用。
- 账户配置变更使用进程内串行写锁和原子文件替换。
- 新增仍使用“连接验证 -> 匿名数据槽 -> 凭据 -> 配置”的补偿事务。
- 更新密码时先写新临时凭据引用，再原子切换配置，最后清理旧引用；失败时旧配置和旧 Supervisor 必须继续可用。

公共能力：

- AccountRuntimeState：starting、ready、syncing、offline、retrying、reauth_required、removing、stopped。
- AccountRuntimeSummary 包含 account_id、state、可选 error_code/retry_at 和 revision。
- 实现添加、获取编辑草稿、更新连接、重新认证、移除影响、移除账户、运行状态列表和最近账户选择命令。
- 事件保持窄边界：accounts-changed、account-runtime-status-changed、account-removing。
- 事件不携带服务器配置、凭据或完整邮件数据。

前端行为：

- 切换账户立即清理文件夹、邮件选择和当前搜索，读取目标账户本地视图并保存最近选择。
- 非当前账户事件只失效该账户缓存，不切换页面、不打断阅读、不显示进度条。
- 账户菜单显示名称、邮箱和简洁运行状态；需要认证时明确提示。
- 设置“账户”分类提供账户列表、添加、同步策略、文件夹映射、编辑、重新认证和移除。
- 添加账户复用首次启动的发现、手动配置、双认证和非加密警告逻辑，不复制两套容易漂移的表单。
- 移除当前账户后安全切换到剩余账户。
- 移除最后一个账户后显示正式“添加账户”状态，但不能重新要求选择数据目录。

- 可在设置中添加至少两个 IMAP/SMTP 密码账户。
- 添加后无需重启即可开始后台同步。
- 切换账户即时显示各自本地邮件，不等待网络。
- 多账户同步、已读/星标/移动/草稿/发件不串数据。
- 一个账户失败不影响其他账户。
- 认证失效与普通网络错误正确区分。
- 编辑配置失败时旧配置继续工作。
- 移除安全停止后台任务并清理本机配置/凭据，但保留匿名本地数据。
- 数据目录、SQLite、事件、日志和前端持久化中不出现账户密码或 Token。
- 单账户 UI 不多出无意义下拉，多账户切换器和设置账户管理是生产级交互。
- 保留现有快速启动、首帧后后台同步和渐进式列表刷新。
