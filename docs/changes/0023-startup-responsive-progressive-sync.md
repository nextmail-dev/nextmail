# 首帧优先与渐进式后台同步

## 问题

此前 Tauri `setup` 在 React 和 WebView 完成首帧之前就启动邮件 Supervisor 与发件 Worker。界面虽然没有在业务逻辑上等待同步完成，但启动同步会立即打开 SQLite、连接 IMAP、下载和解析邮件，与 WebView 首次加载竞争 CPU、磁盘和异步运行时资源。由于同步进度事件也可能早于主工作区监听器，快速完成的同步进入界面后只剩 `complete` 状态，用户看不到收取过程。

## 实现

- Tauri `setup` 仅创建并注册 `AppState`，不再自动启动邮件和发件后台任务。
- 新增幂等 `start_background_services` 命令。主工作区完成一个完整绘制周期后调用该命令；首次设置页面、独立设置窗口和写信窗口不会抢跑同步。
- 邮件 Supervisor 首次启动时立即把当前账户进度写为 `connecting`，随后再进入异步同步。进度保存在 Rust 内存状态中，即使前端错过最早事件也能通过 `get_sync_progress` 读取。
- 发件 Worker 增加启动原子门禁，避免 React Strict Mode 或重复启动通知创建多个发送循环。
- `index.html` 提供不依赖 React、字体或远端资源的浅色/深色中性加载层；React 完成首帧后自动移除，避免 WebView 导航期间出现纯白闪屏。
- IMAP 网络读取继续异步执行；MIME 解析、字符集解码、正文预览、附件分析与 HTML 清洗改由 `spawn_blocking` 执行。当前同步循环逐封等待解析任务，因此同一账户最多提交一个解析任务，不会无界占用 blocking 线程池。
- 文件夹元数据落库后立即发布 `mailbox-changed`；其后每批最多 100 封摘要落库再次发布。前端只失效对应账户/文件夹查询，新数据先入 SQLite，再逐步显示，不等待完整文件夹或完整账户同步结束。

## 行为

1. WebView 首先显示中性加载反馈。
2. React 加载启动配置和本地视图，进入三栏主工作区。
3. 首帧完成后启动 Supervisor 和发件 Worker。
4. 文件夹栏显示启动同步进度。
5. 每个文件夹同步完成后，文件夹计数和邮件列表从 SQLite 渐进刷新。
6. 启动同步完成后进入 IDLE/轮询与静默后台对账；后续自动同步仍不展示进度。

## 验收关注点

- 已配置账户的应用启动时应先出现正式界面和本地邮件，不等待网络连接或首次同步。
- 断网、IMAP 服务器超时或邮件量较大时，主窗口仍应快速可操作。
- 启动同步应显示进度；新文件夹和邮件应在同步过程中逐步出现。
- 完成首次设置后应先进入主界面，再开始首轮邮件同步。
- 重复渲染或重复调用启动命令不得产生多个 Supervisor 或发件 Worker。
- 设置和写信独立窗口不得单独启动第二套后台服务。
