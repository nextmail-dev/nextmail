# ADR 0012：受控桌面通知窗口与定位边界

状态：已接受

日期：2026-07-22

## 背景

第十三阶段需要按全局、账户和文件夹偏好展示新邮件，并提供“层叠”“覆盖”、最大层叠数量和确定展示时间。操作系统通知中心在不同 Windows/macOS 版本上对替换、层叠上限、持续时间和点击回调的控制并不一致，无法可靠表达已经确认的产品行为。

邮件候选来自后台 IMAP Supervisor。首次同步、首次发现文件夹、UIDVALIDITY 重建、正文回填和重复同步不得误报；可见窗口也不能因此获得 SQLite、邮件正文、文件系统、凭据库或通用网络权限。点击后还必须重新核验邮件是否仍在原文件夹，不能把陈旧通知直接当成当前数据库事实。

## 决策

- 使用 NextMail 自有的无装饰 `notification-*` Tauri WebView，而不是伪装成操作系统通知中心。窗口不进入任务栏、显示时不主动抢焦点，并保持在主窗口所在显示器的当前工作区上层。
- `NotificationRuntime` 在 Rust 宿主内拥有窗口队列、覆盖更新、层叠淘汰、超时 generation 和销毁生命周期。React 不创建任意窗口，也不持有计时调度事实。
- 位置按主窗口当前显示器的物理 `work_area` 和缩放因子计算，从右下角向上层叠；实际可见数量还受工作区高度约束，避免窗口落到任务栏或屏幕之外。窗口在定位完成后才显示。
- 覆盖模式复用同一个窗口和不透明通知 ID，并以定向 `notification-content-changed` 事件替换最小 DTO；旧 generation 的计时器不能关闭新内容。层叠模式达到上限时淘汰最早窗口。
- 通知 DTO 只包含不透明通知 ID、公开账户/文件夹/消息 ID、账户显示身份、首个发件人姓名/地址和主题。不包含正文、预览、HTML、附件、原始 EML、内部路径、凭据、Token、服务器错误或任意 URL。
- `notification-*` 使用独立 Capability，只开放 Tauri 事件监听/卸载；窗口通过与自身 label 绑定的 Bootstrap、关闭和激活 Command 以及宿主定向事件交互，不获得文件、网络、数据库、对话框、Shell 或任意建窗权限。临时通知标签由 `window-state` 插件明确排除，不产生可恢复的窗口几何记录。
- 点击时 Rust 先取回当前账户本地视图，核验原文件夹仍可选且邮件位置仍可见，然后聚焦并取消最小化主窗口。有效目标通过定向 `open-mail-location` 事件定位到账户、文件夹和邮件；邮件已移动/删除时只打开原文件夹，文件夹失效时降级到该账户 Inbox 或首个可选文件夹，绝不跨账户猜测。
- 偏好改变或账户移除时关闭受影响的当前通知。应用退出依赖 Tauri 进程统一销毁窗口，不保存临时通知状态。

## 影响

- Windows 上可以确定实现层叠、覆盖、数量和持续时间；代价是通知不会进入系统通知历史，也不能宣称服从操作系统通知中心的聚合或勿扰规则。
- 窗口调度不持有 SQLite 写锁，邮件候选仍只在整次账户同步成功后产生；可见通知失败不会回滚邮件同步。
- 主窗口选择 hook 增加通知定位入口，但最终账户、文件夹和消息状态仍由 TanStack Query 本地视图决定，事件不携带邮件正文。
- Windows 10 22H2+ 与 macOS 12+ 的实机门禁已于 2026-07-22 通过，包括 macOS Spaces、多显示器、工作区、置顶层级和点击聚焦。

## 验证门禁

- Rust 回归覆盖层叠淘汰、覆盖复用、旧计时器失效、账户移除、DPI/工作区定位和屏幕容量钳制。
- 前端回归覆盖最小通知内容、覆盖事件、关闭/激活窄命令、跨账户定位和失效文件夹降级。
- Windows 与 macOS 实机覆盖主窗口普通、最小化和多显示器状态，连续到信的两种模式、数量/时间、手动关闭、点击定位及邮件已移动/删除场景。
