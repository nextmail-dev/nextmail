# ADR 0008：邮件渲染保真度与交互边界

状态：已接受

日期：2026-07-21

## 背景

ADR 0002 确立了“Rust 清洗 + 无权限 sandbox iframe”的安全基线。现有实现因此能够隔离脚本、表单、同源访问和顶层导航，但也整段删除邮件 `<style>` 与所有链接 `href`，并用 `!important` 页面颜色压过一部分作者配色。第十阶段需要提高常见通知、交易和营销邮件的排版保真度，并增加受控外链，不能把不可信邮件放回具备 Tauri IPC 的主 React DOM。

邮件客户端 CSS 兼容性资料用于确定需要保留的展示能力，不代替桌面 WebView 的威胁模型。CSS 也可能产生网络请求、固定遮罩、资源消耗或界面伪装；链接则可能导航到钓鱼站点、本机路径或自定义协议。

## 决策

### 1. 保留不透明 iframe 隔离

- 继续由 Rust 生成完整 HTML 文档，并在不透明 origin、`referrerpolicy="no-referrer"` 的 `srcdoc` iframe 中显示。
- iframe 只允许 `allow-popups`，用于把用户点击的安全链接转换为宿主可拦截的新窗口请求；不增加 `allow-scripts`、`allow-forms`、`allow-same-origin` 或顶层导航权限，邮件文档不能访问主应用 DOM、Tauri IPC、任意文件或通用网络能力。
- 主 WebView 的新窗口处理器始终拒绝在应用内创建外部网页窗口；`allow-popups` 不等于允许邮件自行打开新的 NextMail WebView。
- 不把清洗后的邮件直接插入主 React DOM。专用 WebView 会引入新的窗口、Capability、IPC 和生命周期面，当前没有必要采用。
- 若后续实机证明所选方案无法工作，应停下重新提议 ADR；不能用放开 sandbox 权限作为兼容性降级。

### 2. CSS 采用“解析后重建”的安全子集

- `<style>` 与行内 `style` 必须经过 CSS 语法解析后重建，不能依赖字符串替换或仅检查关键字。
- 允许邮件常用的类型、类、ID、后代/子元素选择器，以及仅接受受限 An+B 参数的 `:nth-child()` / `:nth-last-child()` / `:nth-of-type()` / `:nth-last-of-type()` 结构伪类；不开放 `:has()` 等可查询任意后代关系的函数。响应式宽度和 `prefers-color-scheme` 媒体查询保持受控，声明继续使用展示属性白名单。
- 保留 HTML 邮件仍普遍使用的安全展示属性，包括表格宽高、居中、单元格间距/内边距、对齐、背景色和传统字体属性；不能用阅读器全局字体、内边距、断词或 `max-width` 覆盖原邮件几何布局。
- 丢弃外部样式表、`@import`、`@font-face`、动画与关键帧、生成内容、固定/粘性定位、超大层级、行为属性和解析失败的规则。
- 初始实现丢弃全部 CSS `url()` 与其他资源引用。远程背景、字体或列表资源只有在以后进入统一的不透明资源协议并受现有“默认阻止远程内容”偏好约束后才能启用。
- 对样式表长度、规则数、选择器复杂度和声明长度设置明确预算；超限规则按安全失败方式丢弃，原始 EML 仍是可重建来源。

Rust 锁文件当前已间接包含 `cssparser 0.37`，其许可证为 MPL-2.0。用户在第一批验收时已明确确认第二批可以把它声明为直接依赖并用于安全 CSS 解析；该确认不扩展到其他非宽松许可证依赖。

### 3. 深色模式只提供兜底，不覆盖作者明确样式

- NextMail 为没有显式页面颜色的邮件提供浅色或深灰/浅灰默认值、链接色、分隔线和表格默认色。
- 移除当前 `html/body` 的 blanket `!important`。兜底样式先于邮件作者样式，作者明确的颜色和背景优先。
- 安全保留 `color-scheme`、`supported-color-schemes` 和受控 `prefers-color-scheme` 规则，并用无样式、原生深色与混合背景语料同时验证浅色和深色模式。

### 4. 安全外链直接交给系统关联程序

- Rust 只接受规范化后的 `http`、`https` 和 `mailto`；控制字符、用户信息混淆、本机路径和其他 scheme 均移除。
- 清洗文档直接保留规范化目标，并固定 `target="_blank"` 与 `rel="noopener noreferrer"`；React 不监听点击、不显示离站确认，也没有接受任意 URL 的 Command。
- 主窗口从既有 Tauri 平台配置显式创建并安装 `on_new_window` 处理器。处理器不信任文档值，对目标再次执行相同校验，随后调用只在 Rust 内可用的系统 opener；无论成功、失败或被拒绝，都返回 `NewWindowResponse::Deny`，不在应用内加载目标。
- Windows WebView2 和 macOS WKWebView 的新窗口回调及系统关联行为必须分别实机验证。当前只以 Windows 作为本阶段验收平台；未经执行不得宣称 macOS 通过。

## 隔离方案比较

| 方案 | 结论 | 原因 |
| --- | --- | --- |
| 主 React DOM 内渲染 | 拒绝 | 与具备 Tauri IPC 的应用页面共享 DOM 与权限上下文 |
| sandbox + `allow-same-origin` | 拒绝 | 宿主可重新直接访问邮件 DOM，破坏 ADR 0002 的不透明 origin 边界 |
| sandbox 内注入桥接脚本 | 拒绝 | 重新启用邮件文档脚本执行面，扩大 CSP 与消息校验边界 |
| 专用隔离 WebView | 暂不采用 | 需要新增窗口、Capability、IPC 和生命周期；现有 iframe 能满足展示需求 |
| `allow-popups` + 宿主 `on_new_window` | 采用 | 用户点击可到达宿主；Rust 复验并系统打开，宿主始终拒绝创建应用内网页窗口 |
| `sandbox=""` + Rust 自定义协议 | 放弃 | Windows WebView2 实机点击没有触发可用动作，且增加事件、持久化映射和确认链路 |

## 影响

- ADR 0002 的核心决策继续有效；本 ADR 补充保真 CSS、深色模式和链接交互边界，不允许降低隔离级别。
- 第二批已新增 CSS 解析过滤与缓存失效；第三批修正了 HTML 传统布局属性和外链系统打开。邮件仍处于无脚本、无表单、无同源、无顶层导航和无 IPC 的 iframe 中。
- CSS 背景图、Web Font 和其他网络样式资源暂时可能降级；保留它们不能优先于远程内容默认阻止和本机数据隔离。
- 新窗口回调若在目标 WebView 上无法稳定做到“系统打开并拒绝应用内窗口”，必须回到架构评审，不得开放顶层导航或前端通用 opener。

## 第二批实施记录

- 2026-07-21：经第一批明确确认后，将现有锁文件内的 `cssparser 0.37` 以 `default-features = false` 声明为直接依赖；没有引入新的 crate 或商业/Cloud/Pro 组件。
- `<style>` 和行内 `style` 共用声明白名单与 token 资源检查。只保留受控普通规则、属性选择器、响应式宽度/方向及 `prefers-color-scheme` 媒体查询；丢弃其他 at-rule、网络 URL、未知函数、固定定位/层级/动画等非展示边界。
- 单个样式表、行内样式、选择器、声明值、规则数和嵌套深度均设置预算。CSS 转义解码后重新序列化，并对 `<style>` raw-text 结束序列再次转义。
- 数据格式版本 10 失效旧 `safe_html` 缓存；正文请求优先从账户归属已验证的本地原始 EML 重新清洗，只有本地原文缺失或不可解析时才回到 IMAP。第二批于 2026-07-21 完成 Windows 手动验收。外链桥接仍未实施。

## 第三批实施记录

- 第一版不透明 ID、自定义协议和确认 UI 自动回归通过，但 Windows WebView2 实机点击无动作，未通过验收。根据用户明确反馈，该原型在提交前撤销，不作为最终架构。
- 最终实现直接保存规范化 `href`，iframe 仅增加 `allow-popups`。主 WebView 的 `on_new_window` 再次验证目标、调用 Rust 系统 opener 并始终拒绝应用内窗口创建；React 不参与链接处理。
- 实机截图同时揭示传统邮件布局丢失：清洗器现保留 `class`、`id`、表格宽高/居中/间距/对齐/背景色和传统字体属性，并移除阅读器注入的全局内边距、字体、任意断词及图片/表格最大宽度覆盖。
- 迁移 0011 因可能已应用而保持不可变；0012 删除其临时 `message_links` 表、把格式版本提升到 12，并再次失效旧 HTML 缓存。第三批没有新增依赖、前端 Capability、邮件脚本、表单、same-origin 或顶层导航权限；修正结果已于 2026-07-21 通过 Windows 手动验收，macOS 未执行。

## 第四批实机修正记录

- 一封合成自真实问题结构的发票语料显示：作者把 `tr` 设为 flex container，并用 `th/td:nth-child(...) { flex: ... }` 保证表头与数据行共享 2:2:1:3:1:1 的列宽。旧选择器白名单拒绝所有函数 token，导致整组安全列宽规则被丢弃。
- 选择器解析现只增加上述四种结构伪类，并将参数限制为整数、`n`/`-n`、整数 `n` 维度、`odd`/`even` 与正负号；其他函数、任意嵌套选择器和网络/行为能力仍按失败关闭。迁移 0014 把数据格式提升到 14 并失效旧 HTML 缓存，使阅读器能从本地原始 EML 重建；Composer 新建回复/转发本来就直接读取原始 EML。

## 验证门禁

- 共享语料覆盖无样式正文、普通交易表格、使用 `nth-child()` flex 列宽的发票表格、响应式营销邮件、原生深色、混合背景、普通外链/远程资源和主动恶意内容。
- Rust 回归必须持续移除脚本、事件、表单、嵌入文档、危险 scheme、CSS 网络 URL 与固定遮罩，并保留严格文档 CSP。
- 前端回归必须精确断言 sandbox 只有 `allow-popups`，不含 scripts/forms/same-origin/top-navigation，且保留 `no-referrer`；远程图片只能在现有显式授权后扩展 `img-src`。
- Rust 回归必须验证传统表格属性、class/ID 与受限 `nth-*()` 列宽选择器仍可工作，同时不恢复脚本、事件、危险 URL、任意选择器函数或 CSS 网络资源。
- 2026-07-21 第一、二批 Windows 验收已确认隔离边界、`cssparser 0.37` 选择和 CSS/深色实现。第三批第一次实机结果推翻自定义协议方案并触发上述修订；同日用户确认修正后的直接系统打开与布局保真通过 Windows WebView2 验收。
